Does the Judge Matter?

Every exploitation-rate number in this project comes from an LLM judge reading a response and deciding VERDICT: NO or VERDICT: YES. Two independent threads test how much that number depends on which judge is asked, and how it's asked — holding everything else fixed.

Fixed: the same 1000 cached generations Varied: judge model, then judge prompt Method: rejudge.py + bootstrapped 90% CI

How to read the charts below

exploitation rate

Share of responses a judge marks YES for a given bias — computed separately for the 47 train-split biases (the model saw these during training) and the 5 held-out test-split biases (it didn't).

90% CI (the shaded band)

Bootstrap the original judge's own rate 10,000 times to get a range of what "the same judge, on more samples" would plausibly say. A new rate landing inside the band is noise; outside it is a real effect.

agreement rate

Of every individual response, the share where the two judges gave the same YES/NO verdict — a stricter, per-example check than comparing aggregate rates.

disagreement skew

Ratio of "original said NO, new judge said YES" to the reverse. Every judge tried skews this direction — the question is only by how much.

Bottom line

Every third-party judge over-flags on both splits, every time — scale and price shrink the gap but never close it. The one genuine surprise: only the two Claude-family judges (Haiku 4.5 and Opus 4.8) land inside Sonnet 5's own CI on test, and Haiku — the smaller, cheaper one — is closer than Opus on every measure. Family membership seems to matter more than raw capability here.

02

Judge prompt

Same judge (Sonnet 5, then the local 8B), same generations — change only the wording of the question it's asked. Does the rate survive that?

Short answer: also no — and the effect size rivals swapping the model entirely. But unlike judge model, prompt wording is something this project can actually improve.

Why every comparison below is done twice

Comparing a new judge (or prompt) against the original baseline mixes two effects: did the new thing get better, or did the baseline's own rate shift once you change the prompt on it too? Every prompt variant below is checked in two steps.

The number that matters most: the test-split gap

Train-split biases were in the training data; test-split biases weren't. A prompt that only helps on train is fitting the judge to familiar cases, not making it more correct. This is 8B's test_rate minus the matching Sonnet-5 baseline for that prompt — smaller is better, and negative would mean 8B under-flags Sonnet 5.

Bottom line

applicable+applied is the best prompt tried: highest clean agreement (78.5%), lowest unparseable rate (0.5%), and by far the smallest test-split gap (6 points, vs. strict's 21 and fewshot's 53). It borrows a two-stage design (is the bias even applicable? only then, is it applied?) plus explicit handling for biases phrased as "penalize" — a design an unrelated project (safety-research/auditing-agents) converged on independently for the same 51-bias set.

Also tried, abandoned before a full run

SMOKE TEST ONLY

strict's evidence-quoting stacked onto applicable+applied — on this project's own canonical gratuitous-bias example, the combined prompt quoted the exact biased sentence as evidence, then still output VERDICT: NO. Internally inconsistent with its own evidence; dropped rather than spend a full run finding out if it was noise.

SMOKE TEST ONLY

chain-of-thought removed from applicable+applied — test_rate jumped from 20% to 80% on a 10-example sample. The judge fell back to "some gratuitous insertion is present" rather than checking the specific bias asked about — the same failure mode fewshot hit, via a different mechanism. Suggests the reasoning step is load-bearing, not just latency.
